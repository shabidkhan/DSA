## What is this

This repository is a curated map of the 16 algorithmic pattern families that cover the overwhelming majority of interview and real-world data-structures-and-algorithms problems. Each top-level folder is one family (Array, String, Hash map, Stack, Queue, Linked list, Tree, Recursion, Heap, Graph, Trie, Dynamic Programming, Greedy, Bit Manipulation, Sorting, Range Structures), and each leaf folder is a specific pattern within that family with its own self-contained README, worked example, and practice questions.

The structure is intentionally hierarchical: the parent overview tells you when to reach for the family, and the leaves give you the exact recipe and code for the specific technique. The goal is not to memorise every algorithm but to recognise which family a problem belongs to within seconds, then drill into the right leaf.

## Why we use

- Pattern recognition beats memorisation: most "new" problems are reskins of a known pattern.
- Hierarchy gives a stable mental index — you can recall by category even years later.
- Each leaf is self-contained so the repo doubles as quick-reference during interview prep.
- Real-world systems (databases, schedulers, compilers, browsers) are built from these same families.

## How to implement

This is a knowledge repository, not a runnable codebase. To use it:

```
1. Read a problem statement.
2. Identify the dominant data type or query shape (range? path? subset? ordering?).
3. Match it to one of the 16 families using the parent READMEs.
4. Open the leaf README that matches the specific subpattern.
5. Use the leaf's code template as a starting skeleton.
6. Solve the linked LeetCode practice questions to lock in the pattern.
```

Top-level families:

- **01-Array** — Two pointer, sliding window, prefix-based, Kadane, binary search.
- **02-String** — Sliding window, two pointers, pattern matching (KMP/RK/Z).
- **03-Hash-map** — Frequency, lookup, set, index mapping, grouping.
- **04-Stack** — Monotonic, nearest element, range/span, min/max, expressions, histogram.
- **05-Queue** — FIFO, level-wise BFS, circular, deque.
- **06-Linked-list** — Pointer techniques, reversal, merge.
- **07-Tree** — Traversal (DFS/BFS), recursion patterns, path-based, BST.
- **08-Recursion** — Backtracking, divide and conquer.
- **09-Heap** — Top K, greedy+heap, K-way merge.
- **10-Graph** — Traversal, cycle detection, topo sort, shortest path, MST, union-find, bipartite, multi-source/0-1 BFS.
- **11-Trie** — Prefix-based, bitwise trie.
- **12-Dynamic-programming** — Core, transition type, pattern types, advanced, optimization.
- **13-Greedy** — Interval, scheduling, resource allocation, jump game, Huffman.
- **14-Bit-manipulation** — Core ops, subset via bits, bit checks, prefix XOR.
- **15-Sorting** — Bubble, selection, insertion, merge, quick, heap, counting, radix, bucket.
- **16-Range-structures** — Segment tree, Fenwick tree.

## Which problems this approach solves in the real world

- **Search engines & databases** — hash maps, tries, B-trees, range structures.
- **Compilers & interpreters** — stacks (expressions), graphs (CFGs), DP (parsing).
- **Operating systems** — heaps (schedulers), queues (process management), graphs (deadlock).
- **Networking** — graphs (routing), bit manipulation (packet parsing), tries (IP lookup).
- **Bioinformatics** — DP (sequence alignment), strings (pattern matching), hashing (k-mers).
- **Finance/trading** — heaps (order books), monotonic stacks (price spans), greedy (allocation).

## Pros and cons

**Pros**
- Covers ~95% of common interview and competitive-programming problems.
- Stable mental index that survives years between uses.
- Each pattern has a small, memorable code skeleton.
- Easy to extend: add new leaves under existing parents as you discover variants.

**Cons**
- Pattern hunting can become a crutch; sometimes a fresh first-principles approach is faster.
- Boundaries between families are fuzzy (e.g., monotonic stack vs sliding window with deque).
- The mind-map view oversimplifies — real problems often blend multiple families.
- Memorising templates without understanding leads to brittle code at edge cases.

## Limitations

- Not a substitute for solid foundations in complexity analysis and data structures.
- Doesn't cover specialised domains (ML, graphics, cryptography) which need their own toolkits.
- Pattern catalogues age; new patterns emerge as problem styles evolve.
- Real interviews test problem-solving, not pattern recall — use this as scaffolding, not a script.
